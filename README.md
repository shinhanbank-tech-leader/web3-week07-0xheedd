## 1. 스마트 컨트랙트 업그레이드 방식

### 업그레이드가 어려운 구조
스마트컨트랙트는 탈중앙화, 불변성을 전제로 설계된 블록체인의 프로그램입니다. 이러한 철학에 기반하여 스마트컨트랙트는 업그레이드가 구조적으로 어려우며 디자인 단계부터 업그레이드 가능성을 넣는 패턴이 고안되었습니다. 또한 중앙화된 통제 주체가 없는 환경에서, 업데이트의 정당성을 확보하기 위해 네트워크 참여자 간의 합의를 이끌어내는 온체인 거버넌스 체계가 발전해왔습니다.

- **코드 불변성**: 배포된 컨트랙트 바이트코드 롤백 불가
- **스토리지 충돌 위험**: 스마트컨트랙트의 스토리지는 데이터와 로직이 강하게 결합된 구조로, 업그레이드 중 변수 순서가 틀리면 치명적 오작동 발생

### 프록시 패턴 : 데이터와 로직의 분리
프록시 패턴은 '데이터'와 '로직'을 물리적으로 분리하여, delegatecall을 통해 주소 변경 없이 로직만 교체하는 아키텍처입니다.

- **Proxy(데이터 장부)**: 고객이 접속하는 고정 주소이며, 실제 잔액 등 모든 데이터를 보관합니다.
- **Implementation(로직 코드)**: 실제 계산 로직만 담고 있으며, 업데이트 시 새로운 버전으로 교체됩니다.
- **delegatecall**: "로직은 빌려오되 기록은 내 장부(Proxy)에 하라"는 특수 호출로, 데이터 손실 없는 업그레이드를 가능케 합니다.

**주요 프록시 패턴**
- **Transparent Proxy**: 프록시 컨트랙트 내에 업그레이드 함수를 포함시키며, admin만 호출가능하도록 제어
- **UUPS (EIP-1822)**: 구현 컨트랙트 자체에 업그레이드 함수를 포함시켜 프록시의 가스비를 저렴하게 만듦
- **Beacon Proxy**: 여러 프록시가 beacon 컨트랙트를 바라보도록 지정하여, 로직 주소를 일괄로 업그레이드하는데 사용 (예: 개인지갑 관리 등)
- **Diamond pattern (EIP-2535)**: 컨트랙트를 기능별로 모듈화하여 프록시 내부에 함수-facet 주소를 매핑하여 로직 수행

### 거버넌스: 업그레이드를 허용하는 방법
- **멀티시그(Multisig)**: 여러 키 보유자가 서명해야만 업그레이드 실행. 구성원 중 일부 이상이 서명하면 합의된 것으로 간주하는 구조
- **타임락(Timelock)**: 업그레이드 제안 후 최소 24~48시간 대기 기간 의무화. 사용자가 비동의 시 자산을 빼낼 시간을 보장
- **DAO 거버넌스**: 토큰 홀더 투표로 업그레이드 승인. 다이아몬드 패턴을 활용해 커뮤니티 참여자가 전체가 아닌 특정 기능에 집중하여 선택하도록 설계 가능

## 2. 은행 시스템 변경 방식

### 방향성
은행 시스템은 서비스 중단이 없는 무중단 운영과 거래 데이터의 무결성을 최우선으로 합니다. 블록체인과 달리 '수정 가능한 인프라'를 전제로 하며, 장애 발생 시 즉각적인 롤백이 가능해야 합니다. 또한 모든 변경 사항은 금융당국의 규제와 내부통제 프로세스에 기반하여 제어됩니다.

### 배포전략 : 고가용성 보장과 위험분산
- **피쳐 플래그**: 코드 내부에 조건문(스위치)을 두어 특정 기능을 실시간으로 on/off 제어 가능
- **카나리아 테스트**: 전체 배포 전, 일부 사용자에게만 신규 버전을 우선 노출하여 오류 모니터링 가능
- **블루-그린 배포**: 기존 운영환경(Blue)과 동일한 신규 환경(Green)을 동시에 준비한 후, 검증 완료 시 로드밸런서에서 L4 단위로 트래픽을 한번에 전환

### 거버넌스: 내부통제와 승인프로세스
- **내부통제 및 직무분리**: 직무 분리(SoD), 접근 제어 시스템(IAM) 등 개발/운영/배포 권한을 엄격히 분리
- **변경 관리 프로세스의 표준화**: 변경 요청(RFC), 다단계 승인체계를 마련하여 일반통제(ITGC) 준수
- **사후검증 및 감사**: 감독기관의 감사 아래, 전자금융법, 보안, 장애예방 프로세스 등 규정된 절차와 변경기록 검증

## 3. 은행 IT 관점에서 스마트컨트랙트 업그레이드

인터넷뱅킹, 모바일, 단말 등 채널 시스템은 사용자와 밀접해있어 프로그램 수정이 잦고, 핫픽스와 무중단배포를 적용하는 업무가 많습니다. 상품 안내문구/금리 변경, 약관변경 등의 규모가 작은 수정에도 기존 시스템의 데이터와 자산을 모두 새로운 주소로 마이그레이션해야 한다고 비유하면 기민한 대처는 불가해보입니다. 또한, 원천 데이터와 서비스로직이 강하게 결합되어 있어 기술적 불편함 뿐만 아니라 핫픽스에 대한 작업부담도 훨씬 크게 느껴집니다.

### 스마트컨트랙트 적용시 고려해야할 부분

#### 롤백 및 장애대응
- **서킷 브레이커 설계**: 중대한 오류·취약점 발생 시, 스마트컨트랙트의 특정 기능을 즉시 중단할 수 있는 로직 구현 필요
- **보정 트랜잭션 마련**: 온체인 상태는 롤백이 불가하므로 잘못된 상태를 상쇄하는 상호보완적인 트랜잭션 프로세스 필요
- **스토리지 레이아웃 규정**: 데이터 오염 방지를 위한 레이아웃 규정 마련 및 문서화

#### 금융감독원 전산 규정 준수
- **Multi-sig 구현**: 당행 형상관리 체계처럼 체크인 등록-배포-롤백 다단계 승인 프로세스 구현 필요
- **권한 관리**: admin 권한을 단일키로만 유지하면 SPOF(Single Point of Failure) 위험이 크므로 분산 관리 필수
- **감사 친화성 활용**: 온체인 거버넌스 기록을 은행 내부통제 및 감사 데이터로 매핑하여 활용
